# Skill-to-Course

> **致谢 / Credits**
> 本 skill 的核心思路来源于 [codebase-to-course](https://github.com/zarazhangrui/codebase-to-course)。
> 原项目将代码库转换为交互式课程页面；本 skill 将这一创意迁移到 OpenClaw skill 体系。
> 感谢原作者 [@zarazhangrui](https://github.com/zarazhangrui) 🙏


将任意 OpenClaw skill 转换为**美观、自包含的单页 HTML 课程页面**。让用户在安装 skill 之前，就能真实感知：它能做什么、怎么触发、背后怎么运作。

不是说明书。是**体验**。

---

## 四条设计哲学（生成时必须严格遵守）

这四条是红线，不是建议。每次生成前默读一遍，生成后自查一遍。

### 1. 先体验，后理解（Build first, understand later）

课程的逻辑顺序：
1. 先展示**这个 skill 能帮你做到的事**（结果）
2. 再展示**真实的对话示例**（体验）
3. 最后才解释**背后怎么运作**（原理）

❌ 禁止：先讲原理和架构，再讲"有什么用"
✅ 正确：第一屏就是"你能用它做这件事"，原理在后面

### 2. 演示，而非说教（Show, don't tell）

**每屏至少 50% 是可视化内容**（卡片、流程图、对话气泡、工具链动画、对照表）。

文字块长度不硬性限制，但优先考虑能否可视化——能变成卡片/列表/图示的，尽量可视化，让信息更易读。

检查清单（每个模块生成前过一遍）：
- [ ] 这个内容能变成流程图吗？→ 变
- [ ] 这个内容能变成对话气泡吗？→ 变
- [ ] 这个内容能变成并排对照表吗？→ 变
- [ ] 这个内容能变成卡片网格吗？→ 变
- [ ] 实在不能可视化，才用文字，长度不限但尽量精练

### 3. 测验考实操，不考定义（Quizzes test doing, not knowing）

❌ 禁止出的题型：
- "xxx 的全称是什么？"
- "以下哪个是 skill 的触发词？"
- "这个 skill 支持哪些文件格式？"

✅ 必须是场景题，格式：**"你遇到了[具体情况]，你会怎么做？"**

示例（针对 xlsx skill）：
- ❌ "openpyxl 是什么？"
- ✅ "你的财务模型修改后 =SUM() 显示 0，你下一步应该？"

每道题必须有：选项 + 选后反馈（答对为什么对 / 答错正确思路是什么）

### 4. 比喻要专属，不能复用（No recycled metaphors）

给 skill 内的每个技术概念配一个**独特的、只属于这个概念**的比喻。

规则：
- 每个比喻只用一次，绝不在同一课程里重复
- 禁止使用以下过度滥用的比喻：餐厅/厨师（API）、邮差（消息传递）、工厂流水线（数据处理）
- 比喻要贴合概念的**具体特征**，不是随便一个"大家都懂的"事物
- 好比喻感觉"当然就应该是这个"，而不是"勉强能用"

## 触发方式

用户可以这样说：
- "帮我预览 skill：xlsx"
- "给我看看 citadel skill 能干嘛"
- "把这个 skill 转成课程页面"
- 直接提供 SKILL.md 路径或内容

如未指定 skill：
> "请告诉我要预览哪个 skill — 可以给我 skill 名称、路径、或直接粘贴 SKILL.md 内容。"

## 工作流程

### Phase 1：读取 Skill 内容

**确定 skill 位置**（按优先级）：
1. 用户给的路径
2. `/app/skills/<name>/SKILL.md`
3. `~/.openclaw/skills/<name>/SKILL.md`
4. 用户粘贴的内容

**读取文件**：
- `SKILL.md`（必读）
- `references/` 下所有 .md 文件（如有）
- `scripts/` 下脚本的文件名和头部注释

### Phase 2：分析 + 构思课程

提取：
- **核心能力**（3-7 件事）
- **典型使用场景**（具体的，不是"多种场景"）
- **工作流程**（用户说话 → 每一步 → 最终交付）
- **工具调用链**（调用了哪些工具/API/命令）
- **输入输出对**（2-3 个具体例子）
- **技术术语表**（每个术语配一个通俗解释 + 一个专属比喻）

竞品调研（Phase 2 必须完成，Phase 3 直接用）：
- 用 WebSearch 搜索 skill 所属领域的 2-3 个同类产品/工具
- 对比维度：核心功能、使用方式、自动化程度、独特能力
- 提炼 3 个核心优势（该 skill 相比同类产品做得更好的地方）
- 提出 1-2 条优化建议（以友好口吻指出可以做得更好的方向）

构思比喻（Phase 2 必须完成，Phase 3 直接用）：
- 列出 skill 里所有需要解释的技术概念
- 为每个概念想一个独特比喻，满足"专属性"要求
- 同一课程内不重复

收集获取方式（Phase 2 末尾）：
- 询问用户："是否需要在页面顶部展示 skill 的获取方式？"
- 支持四种素材（可组合）：
  - **链接**：显示为可点击的跳转按钮
  - **图片**：读取并 base64 编码后嵌入 `<img>` 展示
  - **文字描述**：直接显示在获取区
  - **文件（md/zip）**：读取并 base64 编码嵌入 HTML，点按钮触发浏览器下载
- 用户提供素材后，Phase 3 自动填充到页面顶部的获取区
- 如用户不需要，获取区不生成

### Phase 3：生成课程 HTML

读取 `references/html-template.md` 获取完整模板。

输出到：`~/.openclaw/workspace/skill-previews/<skill-name>-course.html`

**重要：生成的页面是正式可用版本，不是演示。禁止在页面中插入任何"演示页面""demo""请在真实环境中使用"等提示文字。**

#### 模块结构（顺序不可调换，体现"先体验后理解"）

| # | 模块 | 核心问题 | 可视化形式 |
|---|---|---|---|
| 0 | 📥 获取这个 Skill | "我怎么安装/下载这个 skill？" | 获取卡片：链接按钮 + 图片展示 + 文字描述 + 文件下载（按用户提供的素材组合） |
| 1 | 🎯 你能用它做什么 | "装了它，我能搞定哪些事？" | 能力卡片网格 + 场景标签 + 竞品对比（知彼知己） |
| 2 | 💬 怎么触发它 | "我怎么说话才能用上它？" | 真实对话气泡动画 + 触发词云 |
| 3 | ⚙️ 幕后发生了什么 | "它做了哪些步骤？" | 流程步骤动画 + 技术/通俗并排对照 |
| 4 | 🔧 用到了哪些工具 | "背后调用了什么？" | 工具链依次出现动画 + 工具说明卡 |
| 5 | 📦 给什么，得到什么 | "具体例子是什么样的？" | 输入→输出示例卡片（2-3 个） |
| 6 | 🧪 试试你掌握了没 | "我真的会用了吗？" | 场景式互动测验（2-3 题）|

#### 强制交互元素（缺一不可）

1. **顶部进度条** — 滚动驱动，实时更新宽度
2. **右侧导航点** — 每章一个点，当前章高亮，hover 显示章节名
3. **流程步骤动画** — 进入视口后依次出现（每步间隔 ~180ms）
4. **工具链动画** — 节点和箭头依次出现
5. **对话气泡动画** — 间隔 ~600-700ms 逐条出现，模拟真实对话节奏
6. **技术/通俗并排对照表** — 左侧代码风格背景，右侧暖白背景
7. **术语 Tooltip** — 所有技术术语悬停显示通俗解释
8. **场景式测验** — 选项点击即时反馈（答对绿色锁定/答错红色可重试 + 文字解释，不揭示正确答案）
9. **Skill 获取区** — 页面顶部，展示下载链接/图片/文字描述/嵌入文件，按用户提供的素材组合显示

#### 设计规范

- **配色**：暖白底 `#FAF8F5`，强调色选珊瑚/靛青/松绿/琥珀之一（绝不用紫色渐变）
- **字体**：标题 Noto Serif SC，正文 Noto Sans SC，代码 JetBrains Mono（Google Fonts CDN）
- **动画**：纯 CSS + IntersectionObserver，进入视口才触发，无外部依赖
- **密度**：文字长度不硬性限制，优先将信息可视化，减少大段纯文字

### Phase 4：交付

1. 告知文件路径
2. 提示打开方式：`open <path>` 或 `xdg-open <path>`
3. 简述各模块内容，邀请反馈

## 质量自查（输出前必须过一遍）

- [ ] 模块 1 是"你能做什么"，不是"这是什么技术"
- [ ] 模块 1 末尾包含竞品对比小节（2-3 个同类产品、3 个核心优势、1-2 条优化建议）
- [ ] 文字内容表达清晰，优先可视化但不强制限制句数
- [ ] 测验题全是场景题，没有定义题
- [ ] 没有重复使用同一个比喻
- [ ] 所有 6 个模块都有明显的可视化主体
- [ ] 所有强制交互元素都存在
- [ ] HTML 文件可以独立在浏览器打开，无报错
- [ ] 如用户提供了获取方式，获取区正确展示在页面顶部（链接/图片/文字/文件按素材组合显示）
- [ ] 如用户未提供获取方式，获取区不生成（不是隐藏）
- [ ] 页面中**绝对不能**出现"这是一个演示页面"或"要创建可编辑版本，请在真实环境中使用"等提示文字——生成的每个页面都是正式版本，不是演示
