算错你的数据？

这个 skill 内置了 9 项数据规范，专门堵这些坑

⭐⭐⭐ 最关键  UV 计算规范——一旦算错，所有转化率都跟着错

❌ 禁止（会低估 40-50%）
COUNT(DISTINCT user_id) / 7.0

把整周用户去重后除以 7——用户可能周一和周五都来，只被算一次，严重低估日均 UV

✅ 正确（每日先去重再平均）
AVG(每日 COUNT DISTINCT user_id)

每天分别算去重用户数，再对 7 天取平均——真实反映每日有多少独立用户看到你的店

其他 8 项必须遵守的规范：

🛡️ 执行前检查清单
日均分母规范 日均单量 = 总订单 ÷ 7（周数据固定除以7，即使某天订单为0也不缩减）
营业状态判断规范 不能用订单数=0判断未营业；正确方式：曝光UV > 0 则视为营业，曝光UV = 0 则视为未营业
订单数据过滤 只统计成单：order_status = 'completed'
门店列表过滤 store_type=1（主站连锁门店）+ store_status=3 + 排除测试店（store_flag='test'）
14列指标全部计算 从日均单量到 UV_CXR 变化，不能缺列
转化率计算准确 UV_CTR = 点击UV ÷ 曝光UV；UV_CXR = 日均订单 ÷ 日均曝光UV
数据必须来自真实查询 禁止编造、近似值或硬编码。所有数字必须来自 Hive SQL 实际执行结果
指标维度标注清晰 标注 PV 还是 UV，日均还是周累计，避免歧义
⚙️ 第四章

一个"门店销量下降"
背后的 6 步诊断

第三部分对每个波动门店的深度分析流程——像医生问诊一样逐层排查

✅ 正确的日均 UV 查询写法（每日去重再平均）：

SELECT AVG(daily_uv) AS avg_daily_uv
FROM (
  SELECT dt, COUNT(DISTINCT user_id) AS daily_uv
  FROM dw_retail.dws_store_flow_entry_daily_di
  WHERE poi_id = '目标门店ID'
    AND dt BETWEEN '2026-03-09' AND '2026-03-15'
  GROUP BY dt
) t -- ✅ 每日先 COUNT DISTINCT，再 AVG

第三步
🌊 流量转化漏斗
曝光UV → 点击UV → 提单UV → 成单，找断层
第四步
广告 vs 自然流量拆分
分别计算两路流量的漏斗，判断波动来源
第五步
🍱 商品维度分析
Top10 商品 WoW 变化，爆款商品单独追

🧪 第六章

3 道场景题——
选出 AI 该怎么做

不考 SQL 语法，考你对分析规范的判断

📊 你让 AI 计算上周日均曝光 UV。AI 写了这条 SQL：
SELECT COUNT(DISTINCT user_id) / 7.0 FROM flow_table WHERE dt BETWEEN '2026-03-09' AND '2026-03-15'

结果跑出来是 1,200。这个数对吗？
A 对，周聚合去重后除以 7，就是日均 UV
B 不对，这样会低估 40-50%。应该每天分别 COUNT DISTINCT，再求 7 天平均
C 不影响，因为 WoW 对比时误差会相互抵消
🏪 某家新门店周一至周三没有订单，周四到周日有订单，共计 40 单。AI 判断：该店本周仅营业 4 天，日均单量 = 40 ÷ 4 = 10 单。这样对吗？
A 对，有订单才算营业，日均单量 = 40÷4 = 10 单
B 不清楚，需要问运营确认门店是否周一就开业了
C 不对。不能用订单数是否为0判断营业。正确：看曝光UV，若曝光UV > 0 则视为营业；日均单量 = 40 ÷ 7 = 5.7 单
🔍 门店销量汇总表出来了，其中某家门店本周日均 8 单，WoW 下降 -30%。另一家门店日均 25 单，WoW 下降 -8%。哪家进入第三部分深度分析？
A 日均 8 单那家，因为下降幅度 -30% 更严重
B 两家都进入深度分析
C 只有日均 25 单那家（满足：日均>10单 且 波动>5%）；8 单那家属于"小门店"，暂不分析
