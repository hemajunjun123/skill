# 知识沉淀 Skill

> 通用知识沉淀能力，支持跨Agent/场景复用。将各类信息结构化提炼后沉淀到知识库中。

---

## 一、Skill概述

- **名称**：知识沉淀
- **触发方式**：斜杠命令（`/沉淀`、`/沉淀 回顾`）、自然语言（"帮我沉淀一下"）、Agent/MCP @调用、对话结束时自动提醒
- **知识库路径**：由首次初始化时确定（见第十一节）
- **运行模式**：
  - **交互模式**：人工触发，提炼后展示确认再入库
  - **静默模式**：Agent自动调用，全自动处理直接入库
  - **对话沉淀模式**：在日常对话中自动识别值得沉淀的业务信息，对话结束时批量处理

---

## 二、输入处理

### 2.1 支持的输入方式

| 输入方式 | 处理方法 |
|----------|----------|
| 对话中直接贴文字 | 直接处理 |
| 本地文件路径 | 读取文件内容后处理 |
| 链接（如钉钉文档） | 尝试读取，无法读取时提示用户用其他方式提供 |
| 图片/截图 | 读取图片内容后处理，提示用户确认提取准确性 |
| PDF | 读取后处理，大文件分页读取 |

### 2.2 输入识别

- 用户**指定了分类**（如"这是一个数据口径"）→ 直接采用，跳过自动分类
- 用户**未指定分类** → 执行自动分类判断

## 三、知识分类体系

### 3.1 四大分类

| 分类 | 定位 | 时效性 |
|------|------|--------|
| **业务知识** | 业务逻辑、流程规范、行业认知、决策记录 | 长期稳定 |
| **策略沉淀** | 成功案例、失败经验、方法论、策略分析、数据洞察 | 阶段性，可能被新认知替代 |
| **数据口径** | 指标定义、计算公式、数据来源 | 相对稳定，变更需明确记录 |
| **会议纪要** | 原始讨论记录 | 原始素材，关键结论需提炼归入其他三类 |

### 3.2 内容性质判断（分类前置步骤）

对输入内容逐段判断其性质，决定是否值得沉淀以及如何提炼：

| 内容性质 | 确定性 | 沉淀策略 | 示例 |
|----------|--------|----------|------|
| **已发生的事实** | 高 | 全面提炼 | 实验数据结果、线上表现、已执行的决策 |
| **分析与推理** | 中 | 重点提炼 | 问题诊断、归因逻辑、tradeoff分析 |
| **计划与意图** | 低 | 仅提炼思路和逻辑，过滤具体工作项、排期、优先级 | 规划、方案中的TODO列表 |

特殊规则：
- **目标不独立沉淀**：目标（如"日增14万AAC"）是动态的，应作为策略沉淀条目的背景字段，不单独成条
- 同一份文档中可能同时包含多种性质的内容，逐段判断而非整体判断

### 3.3 自动分类判断规则（按优先级顺序）

```
1. 先看内容形态
   → 有明确的会议时间 + 参与人 + 讨论过程 → 会议纪要

2. 再看是否涉及指标计算
   → 有公式、分子/分母、数据源定义 → 数据口径

3. 再看时效性
   → 长期稳定的规则/流程/定义/决策 → 业务知识
   → 阶段性的、跟实验/数据相关的经验结论 → 策略沉淀

4. 兜底规则：如果经过以上三步仍无法确定分类 → 默认归为「策略沉淀」
   理由：策略沉淀有有效期（90天）自动提醒review，即使分类有误也不会长期误导；
         而业务知识无有效期，误分入业务知识的阶段性结论会长期存在不被review。
   ⚠ 交互模式下，分类不确定时必须在展示确认中标注「⚠ 分类不确定，默认策略沉淀，可调整」
```

### 3.4 模糊场景判断指南

| 场景 | 判断 | 理由 |
|------|------|------|
| "ROI必须大于1" | 业务知识 | 长期约束规则，不是指标计算 |
| "高价值用户对5元红包不敏感" | 策略沉淀 | 阶段性结论，可能被新实验推翻 |
| "△AAC = 实验组AAC - 对照组AAC" | 数据口径 | 指标计算公式 |
| "会上决定用Thompson Sampling" | 先入会议纪要，再提炼"TS算法决策"入业务知识 | 会议记录+决策 |

## 四、处理流程

### 4.1 交互模式（人工触发）

```
Step 1: 接收输入内容

Step 2: 快速扫描，输出「拆分计划」（不展开正文，节省上下文）
        - 逐段扫描输入内容，识别独立的知识单元（详见 4.4 拆分规则）
        - 如果输入来自文件，记录文件路径；如果是对话文字，按原文段落标记位置
        - 输出拆分计划，格式为简洁列表（必须包含原文定位锚点）：
          1. {标题} → {分类} · {子标签} | 原文位置：{文件名 第X-Y行 / 对话中第X段}
          2. {标题} → {分类} · {子标签} | 原文位置：{文件名 第X-Y行 / 对话中第X段}
          ...
          共 N 条，将分 X 批处理（每批最多 3 条）。确认拆分方案？(Y/调整)
        - 如果文档中有全局背景、术语定义等被多条引用的内容，额外输出：
          📌 共享上下文：{一句话概括，如"文档开头定义了AAC=年度活跃消费者"}
          后续每批处理时会自动带上此共享上下文
        - 等待用户确认拆分方案后，再进入逐批处理

Step 3: 分批处理（每批最多 3 条，处理完一批再处理下一批）
        对当前批次的每条知识执行以下子步骤：
        3a. 根据 Step 2 记录的原文定位锚点，精确读取该条对应的原文段落
            ⚠ 仅读取该条相关的原文段落，不要把整份文档重新加载
            ⚠ 如果 Step 2 标注了共享上下文，将其作为背景附加到当前条目的提炼中
            按分类模板提炼结构化内容
        3b. 轻量重复检测（读目标分类 _summary.md 匹配）
        3c. 按第十二节格式展示该批次的提炼结果，等待用户确认
            - 每条包含：分类·子标签·关联项目、采纳程度、标题、结构化正文、重复提示（如有）
            - 等待用户回复：确认(Y) / 修改意见 / 跳过某条
        3d. 按用户确认结果入库（严格执行！）
            ⚠ 入库文件必须严格按照第六节的「强制文件模板」生成，逐字段填写，禁止省略或自由发挥格式
            ⚠ 每条知识必须独立一个文件，文件名格式：YYYY-MM-DD-标题.md
            ⚠ 文件必须存放在对应分类子目录下（业务知识/、策略沉淀/、数据口径/、会议纪要/）
        3e. 该批次入库完成后，更新 _summary.md 和 _tags.md
        3f. 提示用户：「第 X 批已完成（N/总数），继续下一批？」
            → 用户确认后处理下一批，重复 3a~3f
            → 直到所有批次处理完毕

Step 4: 全部批次完成后
        - 如果用户已配置知识库 MCP Server（kb-search），调用 reindex_knowledge 工具触发增量索引更新
        - 展示本次沉淀汇总：共入库 N 条，跳过 M 条
```

> **上下文管理原则：**
> - 拆分阶段只输出标题列表，不展开任何正文内容
> - 提炼阶段每次只处理 3 条，处理完入库后释放上下文再处理下一批
> - 如果输入只有 1-3 条知识，无需分批，直接走完整流程
> - 禁止一次性把所有条目的完整内容都展开在对话中

### 4.2 静默模式（Agent/MCP调用）

```
Step 1: 接收输入内容
Step 2: 快速扫描，生成拆分计划（同交互模式，但不展示给用户）
Step 3: 分批处理（每批最多 3 条）
        对当前批次的每条知识执行：
        3a. 回到原文中定位该条对应的段落，按分类模板提炼
        3b. 轻量重复检测
        3c. 跳过确认，直接按第六节模板入库
        3d. 该批次入库完成后，更新 _summary.md 和 _tags.md
        3e. 处理下一批，重复 3a~3d 直到全部完成
Step 4: 全部批次完成后
        - 如果用户已配置知识库 MCP Server（kb-search），调用 reindex_knowledge 工具触发增量索引更新
        - 记录本次入库信息，用于定期摘要review
```

### 4.3 会议纪要特殊处理

会议纪要入库后，额外触发拆分提炼：
- 整段记录 → 入 `会议纪要/`
- 提炼出的决策 → 入 `业务知识/`
- 提炼出的口径 → 入 `数据口径/`
- 提炼出的经验结论 → 入 `策略沉淀/`

### 4.4 拆分规则（Step 3 详细指引）

> **核心原则：一条知识 = 一个独立可检索的知识单元。** 宁可多拆不要少拆。

#### 4.4.1 必须拆分的场景

| 场景 | 判断依据 | 拆分方式 |
|------|----------|----------|
| **多个实验/AB测试** | 不同实验变量、不同实验组 | 每个实验独立一条 |
| **多个指标/口径** | 不同指标名称、不同计算公式 | 每个口径独立一条 |
| **多个策略/方案** | 不同策略方向、不同人群/场景 | 每个策略独立一条 |
| **多个决策/结论** | 独立的判断，互不依赖 | 每个决策独立一条 |
| **混合分类内容** | 内容同时涉及多个分类（如既有口径又有策略） | 按分类拆分 |
| **多份文档/多个文件** | 用户一次提供了多份文档 | 每份文档至少独立处理，文档内部再判断是否需进一步拆分 |
| **长文档（超过500字）** | 篇幅较长，包含多个段落/章节 | 按主题段落拆分，每个独立主题一条 |

#### 4.4.2 不需要拆分的场景

| 场景 | 理由 |
|------|------|
| 同一实验的多组数据（如对照组vs实验组） | 属于同一实验的不同维度，合在一起更完整 |
| 同一指标的分子/分母/计算逻辑 | 属于一个口径定义，拆开无法独立理解 |
| 同一策略的背景+做法+效果+结论 | 属于一条完整的策略沉淀 |

#### 4.4.3 拆分执行流程

```
1. 通读全部输入内容
2. 识别独立知识单元的边界（按上表判断）
3. 为每个知识单元独立执行：
   - 分类判断（不同单元可能属于不同分类）
   - 模板提炼
   - 重复检测
4. 所有条目一起展示确认（按第十二节格式，标注"条目 1/N"）
5. 用户可以逐条确认/修改/跳过
```

#### 4.4.4 拆分示例

**输入：** 一份包含3个实验结论 + 1个口径定义的文档

**拆分结果：**
- 条目 1/4：策略沉淀 — 实验A的结论
- 条目 2/4：策略沉淀 — 实验B的结论
- 条目 3/4：策略沉淀 — 实验C的结论
- 条目 4/4：数据口径 — 文档中定义的指标口径

## 五、重复/冲突处理

> 详细的冲突检测与知识关系处理逻辑见第十五节。本节为快速参考。

### 5.1 检测方式

- 读取目标分类的 `_summary.md` 进行匹配
- 如已配置 kb-search MCP Server，同时用语义搜索辅助检测
- 仅在标题/主题高度相似时触发，模糊的不打扰

### 5.2 四种关系类型（详见第十五节）

| 关系类型 | 判断 | 处理 |
|----------|------|------|
| 补充型 | 同主题，新增新维度 | 合并到已有条目 |
| 不同前提 | 同主题，不同适用条件 | 两条共存，互相引用 |
| 同前提更新 | 同前提，新结论替代旧结论 | 覆盖已有条目，旧内容入修订记录 |
| 真实冲突 | 同前提，结论矛盾 | 两条共存，标记冲突，提醒用户判断 |

### 5.3 修订记录格式

```markdown
> 修订记录：
> - 2026-04-08：更新为方案B（原方案A因XX原因调整）
> - 2026-03-15：初始版本，采用方案A
```

## 六、强制文件模板（必须严格遵守）

> **硬约束：入库文件必须严格按照以下模板逐字段填写。禁止省略任何字段、禁止修改字段名、禁止添加模板之外的字段、禁止自由发挥格式。**
>
> **填写原则：**
> - 允许对原文进行合理的逻辑整理和推理，但**数据相关内容（数字、指标、比例、金额）必须严格来自原文，绝对不可编造**
> - 如果原文中没有提供某字段的信息，填写"[待补充]"，不要猜测
> - 关联项目：仅当原文中明确提及项目名时才填写，否则填"通用"
> - 子标签：必须先读取 `_tags.md`，从已有标签中选择，不自行创造新标签（如确需新增，交互模式下先询问用户确认）
>
> **⚠ 数据零容忍 vs 逻辑可推理：**
>
> 提炼内容分为两类，适用不同约束：
>
> | 内容类型 | 约束等级 | 规则 | 示例 |
> |----------|----------|------|------|
> | **数据/数字** | 🔴 零容忍 | 所有数字、指标值、比例、金额、日期等必须原文有才能写，不可编造、推算、四舍五入 | 原文"12.3%"不可写成"约12%"，原文无ROI不可补一个数字 |
> | **逻辑/结论** | 🟡 允许推理 | 可基于原文信息做合理归纳、对比、因果推理，但推理部分需用语气词标识不确定性 | 原文说"5元转化率12.3%，10元转化率8.5%"，可以归纳为"5元转化率高于10元" |
>
> **允许的操作：**
> - ✅ 对原文进行结构化整理（拆句、重新组织段落顺序、去除口语化表达）
> - ✅ 提炼原文的核心信息（从长段落中提取关键结论）
> - ✅ 基于原文已有信息做逻辑归纳（如原文给了A>B和B>C，可以推出A>C）
> - ✅ 对原文中多条信息做对比整理（如将散落在各处的实验结果汇总对比）
> - ✅ 补充合理的上下文衔接（如"因此"、"相比之下"等逻辑连接词）
>
> **禁止的操作：**
> - ❌ 编造任何数字（转化率、ROI、金额、人数、日期等），原文没有的数字一个都不能写
> - ❌ 对原文数字做二次计算（如用两个原文数字相除得出一个原文没有的比率），除非用户明确要求
> - ❌ 添加原文中完全没有提及的外部知识（如引入行业均值、竞品数据）
> - ❌ 将原文的不确定表述改为确定表述（如"可能有效"不可改为"有效"）
>
> **正例/反例：**
>
> | 原文内容 | ✅ 正确提炼 | ❌ 错误提炼 | 错误原因 |
> |----------|------------|-----------|----------|
> | "5元红包转化率12.3%，10元红包转化率8.5%" | 5元红包转化率（12.3%）高于10元红包（8.5%） | 5元红包转化率12.3%，高于行业均值8% | 编造了"行业均值8%"这个数字 |
> | "高价值用户对小额红包不敏感" | 高价值用户对小额红包不敏感，可能需要差异化权益策略 | 高价值用户对小额红包不敏感，建议改用50元以上大额券 | 编造了"50元以上"这个数字 |
> | 原文未提及数据来源 | 数据来源：[待补充] | 数据来源：odps数据仓库 | 编造了数据来源 |
> | "决定用TS算法，之前用的UCB效果不好" | 采用TS算法替代UCB（原UCB效果不佳） | 采用TS算法替代UCB，预计提升30% | 编造了"30%"这个数字 |

## 十三、业务信息识别规则

> 本节定义什么信息值得沉淀，用于对话沉淀模式中的自动识别，也适用于交互/静默模式的输入筛选。

### 13.1 值得沉淀的信息

| 类型 | 说明 | 沉淀方式 | 示例 |
|------|------|----------|------|
| **数据结论**（含背景） | 有明确数据支撑的结论，必须连同背景和策略一起沉淀 | 全面提炼 | "5元红包组转化率12.3%，10元组ROI 1:4.5" + 实验背景 |
| **分析推理** | Claude 或人类基于事实做出的逻辑分析 | 重点提炼 | "高价值用户对小额红包不敏感，可能因为..." |
| **经验判断** | 人类基于经验、直觉或对物理世界的认知做出的判断（非确定结论） | 仅作参考，标注「经验判断」 | "我觉得这个人群用积分比红包好" |
| **决策与共识** | 明确做出的决定、达成的共识 | 全面提炼 | "决定用Thompson Sampling替代UCB" |
| **业务规则/约束** | 长期有效的业务逻辑、流程、约束条件 | 全面提炼 | "ROI必须大于1" |

### 13.2 不沉淀的信息

| 类型 | 理由 | 示例 |
|------|------|------|
| **裸数字/单一数值** | 没有背景的数字无法独立理解 | "ROI是1.2"（无上下文） |
| **原始数据/数据表** | 应存在数据平台，不属于知识 | 大段SQL查询结果、CSV数据 |
| **代码/技术实现** | 应存在代码仓库，不属于业务知识 | 具体代码片段、技术配置 |
| **临时沟通/寒暄** | 无知识价值 | "好的"、"收到"、"明天聊" |
| **重复已有信息** | 避免冗余 | 与知识库现有条目完全一致的内容 |
| **未定义的TODO/排期** | 计划性质，易过期，不属于知识 | "下周做AB测试"、"优先级P1" |

### 13.3 边界判断

- **数据结论** vs **裸数字**：有"因为X所以Y"的逻辑链 → 数据结论；只有"X是Y" → 裸数字
- **经验判断** vs **闲聊**：对业务决策有参考价值 → 经验判断；纯粹社交性表达 → 不沉淀
- **决策** vs **计划**：已明确"决定做X" → 决策；"打算做X"/"考虑做X" → 仅提炼思路

**⚠ 识别偏好：宁可漏掉，不要错标。** 不确定是否值得沉淀时，不标记为候选。
理由：错误沉淀会污染知识库且清理成本高，而遗漏的信息用户可以随时用 `/沉淀` 手动补录。

#### 13.3.1 边界判断示例

| 对话内容 | 是否沉淀 | 理由 |
|----------|----------|------|
| "这轮实验5元红包组转化率12.3%，比对照组高了4个点" | ✅ 沉淀 | 有数据+对比基准+结论 |
| "转化率12.3%" | ❌ 不沉淀 | 裸数字，缺少实验背景 |
| "我觉得这个人群用积分比红包好，之前做过类似的" | ✅ 沉淀（标注经验判断） | 有业务参考价值，基于过往经验 |
| "我觉得今天天气不错" | ❌ 不沉淀 | 无业务价值 |
| "我们决定Q2开始用TS算法" | ✅ 沉淀 | 明确决策 |
| "要不要试试TS算法？" | ❌ 不沉淀 | 提问/讨论中，未形成决策 |
| "帮我查一下昨天的数据" | ❌ 不沉淀 | 操作指令，非业务知识 |
| "上次那个实验结果你还记得吗" | ❌ 不沉淀 | 提问，无新信息 |

## 十四、对话沉淀模式

> 用于个人日常使用场景。在对话过程中自动识别值得沉淀的业务信息，对话结束时批量处理。

### 14.1 工作原理

```
对话过程中：
  - 持续监听对话内容
  - 识别到符合第十三节规则的业务信息时，在内部标记为"待沉淀候选"
  - 不打断对话，不实时处理，不做任何提示

收束时刻（对话自然结束时）：
  - 用户主动说"结束了"/"先这样"等结束语
  - 用户调用 `/沉淀 回顾`
  - 对话即将结束时，主动提醒用户是否需要沉淀本轮对话中的业务信息
```

### 14.2 收束时刻识别

**⚠ 核心原则：不确定时不触发。** 误触发会打断用户思路，远比漏触发更有害。
用户随时可以用 `/沉淀 回顾` 主动触发，所以漏触发的代价很低。

以下信号表明一个讨论线索已收束，适合沉淀：

| 信号类型 | 典型表达 | 说明 |
|----------|----------|------|
| **总结性发言** | "所以结论是..."、"总结一下..." | 最佳沉淀时机，上下文最完整 |
| **话题切换** | "另外一个事情..."、"换个话题" | 上一个话题收束 |
| **确认性回复** | "好的就这样"、"可以，按这个来" | 决策已达成 |
| **对话结束** | "先这样"、"今天到这"、"拜拜" | 整体对话收束 |

**不应触发的场景（防止误判）：**

| 容易误判的表达 | 实际含义 | 为什么不触发 |
|---------------|----------|-------------|
| "好的" | 简单应答 | 不一定是决策确认，可能只是"我听到了" |
| "可以" | 可能是同意继续讨论 | 需要结合上下文判断是确认决策还是允许继续 |
| "我想想" | 思考中 | 话题未收束，用户还在考虑 |
| "先看看数据再说" | 暂停但未收束 | 等待数据验证，结论未形成 |

**误触发恢复：** 如果提醒沉淀后用户回复"不用"/"现在不沉淀"等拒绝表达，立即停止沉淀流程，不再追问，继续正常对话。

### 14.3 处理流程

```
Step 1: 对话进行中 — 标记候选
        - 每轮对话后，判断是否产生了符合第十三节规则的业务信息
        - 如果有，将该轮对话的位置和摘要记录到内部候选列表
        - 格式：{轮次, 摘要, 预判分类, 预判标题}
        - ⚠ 仅记录位置和摘要，不展开提炼，不打断对话

Step 2: 收束时刻触发 — 批量处理
        - 当识别到收束信号，或用户调用 `/沉淀 回顾`
        - 展示候选列表，格式：
          📋 本轮对话中发现 N 条值得沉淀的业务信息：
          1. {预判标题} → {预判分类} | 来自对话第 X 轮
          2. ...
          需要沉淀吗？(Y/调整/跳过)
        - 用户确认后，进入标准交互模式的 Step 3（分批处理）
        - 提炼时回溯对应对话轮次的完整上下文

Step 3: 入库
        - 与交互模式完全一致（模板、确认、更新索引）
```

### 14.4 CLAUDE.md 配置建议

在用户的 `CLAUDE.md` 或 `.claude/settings.json` 中添加以下提示，让 Claude 在长对话中主动提醒：

```
当对话超过 20 轮且包含业务讨论时，在对话结束前主动提醒：
"本轮对话中有 N 条信息可能值得沉淀到知识库，需要执行 /沉淀 回顾 吗？"
```

### 14.5 `/沉淀 回顾` 命令

**触发方式**：用户在对话中任何时刻输入 `/沉淀 回顾`

**行为**：
1. 回溯当前对话的全部历史
2. 按第十三节规则扫描，识别所有值得沉淀的业务信息
3. 输出候选列表（同 14.3 Step 2 格式）
4. 用户确认后执行沉淀

**适用场景**：
- 长对话结束时的批量回顾
- 用户忘记了中途想沉淀的内容
- 对话中途想检查一下已有哪些值得沉淀的信息

## 十五、冲突检测与知识关系

> 替代简单的"重复/覆盖"逻辑，精确识别新旧知识之间的关系类型，采取不同处理策略。

### 15.1 四种关系类型

| 关系类型 | 定义 | 判断依据 | 处理策略 |
|----------|------|----------|----------|
| **补充型** | 新知识为已有知识增加新维度/新数据 | 同一主题，但新内容覆盖了未涉及的方面 | 合并：将新内容追加到已有条目的对应字段中 |
| **不同前提** | 新旧结论不同，但适用前提/条件不同 | 同一主题，但前提条件（人群/场景/时间段）不同 | 共存：两条都保留，互相在"相关知识"中引用 |
| **同前提更新** | 同一前提下，新结论替代旧结论 | 前提条件相同，但结论/数据更新了 | 更新：用新内容替换旧内容，旧内容记入修订记录 |
| **真实冲突** | 同一前提下，新旧结论矛盾且无法确定哪个正确 | 前提相同，结论矛盾，缺乏足够证据判断 | 标记：两条都保留，标注冲突关系，提醒用户判断 |

### 15.2 检测流程

```
1. 新条目提炼完成后，读取目标分类的 _summary.md
2. 找到标题/主题高度相似的已有条目
3. 如果无相似条目 → 直接入库（新增）
4. 如果有相似条目 → 读取已有条目全文，进行关系判断：
   a. 对比前提/适用条件：
      - 前提不同 → 「不同前提」
      - 前提相同或无明确前提 → 继续判断
   b. 对比结论：
      - 结论一致，内容有新增 → 「补充型」
      - 结论不同，有明确新证据 → 「同前提更新」
      - 结论矛盾，无法判断 → 「真实冲突」

⚠ 兜底规则：不确定关系类型时，默认判定为「补充型」（合并）。
  理由：补充型是破坏性最小的操作，最多导致信息冗余，不会丢失旧知识。
```

#### 15.2.1 关系判断示例

| 已有条目 | 新条目 | 正确判断 | 判断依据 |
|----------|--------|----------|----------|
| "5元红包新客转化率12.3%（全量人群）" | "5元红包高价值用户转化率仅3%" | **不同前提** | 人群不同：全量 vs 高价值 |
| "5元红包新客转化率12.3%" | "5元红包新客转化率提升至15.1%（Q2数据）" | **同前提更新** | 同一人群同一指标，Q2新数据替代旧数据 |
| "5元红包新客转化率12.3%" | "5元红包+签到任务组合，转化率18.5%" | **补充型** | 同主题新增了组合策略维度 |
| "建议用Thompson Sampling" | "UCB在冷启动阶段优于TS" | **不同前提** | 场景不同：通用场景 vs 冷启动阶段 |
| "ROI红线1.2" | "ROI红线调整为1.44" | **同前提更新** | 同一约束，数值更新 |
| "TS算法收敛快" | "TS算法收敛慢于UCB" | **真实冲突** | 同前提，结论矛盾，需用户判断 |

#### 15.2.2 易混淆场景辨析

| 容易混淆的情况 | 区分方法 |
|---------------|----------|
| 补充型 vs 不同前提 | 补充型：新内容不改变旧结论，只增加信息量。不同前提：新旧结论可能不同，是因为条件不同导致 |
| 同前提更新 vs 真实冲突 | 同前提更新：有明确证据（如更新的实验数据、新的决策文件）表明新结论取代旧结论。真实冲突：无法确定谁对谁错（如两个实验结论矛盾但都没有明显问题） |
| 不同前提 vs 同前提更新 | 关键看前提是否变了：如果前提变了（人群、场景、时间段明确不同），是不同前提；如果前提一样只是结论变了，是同前提更新 |

### 15.3 各类型处理细节

**补充型 — 合并**
```markdown
- 将新内容追加到已有条目的对应段落
- 修订记录追加："YYYY-MM-DD：补充 {新增内容摘要}"
- 不创建新文件
```

**不同前提 — 共存**
```markdown
- 新条目独立入库
- 在新旧两条的 frontmatter "相关知识" 字段中互相引用
- 在展示确认时提示："与 {已有条目标题} 为不同前提下的结论，将分别保留"
```

**同前提更新 — 覆盖**
```markdown
- 用新内容替换已有条目的正文
- 修订记录追加："YYYY-MM-DD：更新为 {新结论摘要}（原 {旧结论摘要}，因 {变更原因}）"
- 不创建新文件
```

**真实冲突 — 标记**
```markdown
- 新条目独立入库
- 在新旧两条的 frontmatter 中添加：冲突标记: "与 {对方文件名} 结论矛盾，待确认"
- 交互模式下立即提示用户判断
- 静默模式下在定期摘要中高亮标注
```

### 15.4 交互模式展示

当检测到关系时，在确认展示格式中追加：

```
⚠ 与已有条目相关：{已有条目标题}
  关系：{补充型/不同前提/同前提更新/真实冲突}
  建议：{合并/共存/覆盖/标记冲突}
  处理方式？(按建议执行/手动选择)
```

## 十六、知识老化与时效管理

> 知识有时效性，策略沉淀和数据口径尤其容易过期。本节定义主动提醒机制。

### 16.1 有效期标注

在入库时，根据分类自动标注建议有效期：

| 分类 | 默认有效期 | 理由 |
|------|-----------|------|
| 业务知识 | 无限期 | 业务规则通常长期稳定 |
| 策略沉淀 | 90天 | 策略结论可能被新实验推翻 |
| 数据口径 | 180天 | 口径相对稳定，但需定期确认 |
| 会议纪要 | 无限期 | 原始记录，不存在过期问题 |

在 frontmatter 中添加可选字段：

```yaml
有效期: "2026-07-08"  # 入库日期 + 默认有效期，可由用户手动调整
```

### 16.2 老化提醒触发

当以下场景发生时，检查并提醒可能过期的知识：

| 触发场景 | 检查方式 |
|----------|----------|
| 搜索知识库返回结果时 | 检查结果中的条目是否已超过有效期 |
| 沉淀新条目到同一分类时 | 扫描同分类下是否有超期条目 |
| 用户执行 `/沉淀 回顾` 时 | 附带展示当前超期条目列表 |

### 16.3 提醒格式

```
⏰ 以下知识条目可能已过期，建议确认是否仍然有效：
  1. {标题} | {分类} | 入库于 {日期}，已超过建议有效期 {N} 天
     → 选择：仍然有效 / 已过期归档 / 需要更新
```

### 16.4 用户操作

| 操作 | 效果 |
|------|------|
| 仍然有效 | 有效期顺延（重置为当前日期 + 默认有效期） |
| 已过期归档 | 状态改为"已归档" |
| 需要更新 | 提示用户提供新信息，走更新流程 |

## 十七、自动关联机制

> 新知识入库时，自动搜索已有知识库，填充 frontmatter 中的"相关知识"字段。

### 17.1 触发时机

每条知识入库前（确认后、写入文件前），自动执行关联检索。

### 17.2 关联检索逻辑

```
1. 用当前条目的标题 + 关键内容作为查询
2. 如果已配置 kb-search MCP Server → 调用 search_knowledge 语义搜索
   如果未配置 → 读取各分类 _summary.md 做关键词匹配
3. 筛选条件：
   - 相似度 > 0.7（语义搜索）或标题关键词匹配度 > 50%
   - 排除自身
   - 最多关联 5 条
4. 将匹配到的条目文件名写入 frontmatter 的"相关知识"字段
5. 同时更新被关联条目的"相关知识"字段（双向关联）
```

### 17.3 交互模式展示

在确认展示中追加关联信息：

```
🔗 自动关联：
  - {已有条目1标题}（相似度 0.85）
  - {已有条目2标题}（相似度 0.72）
  确认关联？(Y/移除某条)
```

### 17.4 静默模式

自动关联，不展示确认。在定期摘要中标注新增的关联关系。
